## Overview

Extends the v1.0 MVP transcription pipeline with SRT subtitle output and pyannote-audio speaker diarization. Every `meet transcribe` invocation now writes both `.txt` and `.srt` files. When a HuggingFace token and pyannote.audio are available, the transcript is diarized with `SPEAKER_XX:` prefixes; graceful fallback preserves undiarized output otherwise. `meet summarize` automatically prefers the diarized transcript. Three new health checks cover pyannote availability.

## Milestone Summary

**Key Decisions:**

- `transcribe_audio()` returns `(text, segments)` tuple — breaking change to callers fixed in same plan
- SRT always written alongside `.txt` with no opt-out flag
- `generate_srt()` accepts optional `speaker_map` dict keyed by 0-based segment index
- `HuggingFaceTokenCheck` returns WARNING (not ERROR) — diarization is optional
- `PyannoteCheck` returns ERROR when pyannote.audio not importable
- `run_diarization()` lazy-imports `pyannote.audio.Pipeline` — avoids import-time cost
- Diarized content overwrites plain `.txt`; `diarized_transcript_path` points to same file
- `assign_speakers_to_segments` uses max-overlap strategy
- torchaudio.list_audio_backends monkey-patch for torchaudio≥2.9 compatibility
- pyannote.audio version relaxed to `>=3.3.2,<5` to avoid pip dependency conflicts
- `meet init --update` flag uses early-exit routing before interactive config-exists check

**Issues Resolved:**

- pyannote `ModuleNotFoundError` in torchaudio≥2.9 — fixed via `torchaudio.list_audio_backends` monkey-patch
- Pre-existing test_init.py failures from `cb3a91a` commit — fixed in gap closure plan

**Technical Debt Incurred:**

- pyannote.audio 4.x API untested (version pin relaxed to `<5` — may need updates)
- Wave 0 stubs partially converted to real tests (full Nyquist coverage deferred)
